# Phase 05 — Employee Management (PC) — Context

## Goal

Managers can view, create, edit, and change employee status from the PC admin screen.
Requirement: **PC-04**.

---

## Decisions

### D-01 — Navigation Placement
**Decision:** Add "Empleados" as a **5th card** inside the existing "Gestión de datos" card grid (2×3 layout).
- No new top-level sidebar button
- `data.fxml` grid goes from 2×2 → 2×3

### D-02 — "Cambiar PIN" Meaning
**Decision (revised — out-of-phase temp password flow implemented):**
- Employee creation: no manual password in the ADD form — backend auto-generates `TmpXXXXXX` and sets `mustChangePassword=true`. The temp password is returned in the response and shown to the manager in a dialog ("Contraseña temporal: TmpXXXXXX — Dísela al empleado").
- Separate `POST /employees/{id}/reset-password` endpoint exists to reset to a new temp password.

### D-03 — Status Management
**Decision:** All **three** statuses are visible/settable from PC:
- ACTIVO, INACTIVO, SUSPENDIDO
- Changed via **ComboBox** inside the EDIT form (not a separate button)

### D-04 — Role Assignment UX
**Decision:** **Toggle buttons** for each available role (same pattern as DataOperationController uses for other toggle actions).
- Roles loaded from backend at form init
- Multi-role selection allowed (Set<Role>)

### D-05 — Screen Architecture
**Decision:** **Extend DataOperationController** — add `EMPLOYEE` to `EntityType` enum.
- Avoids a new controller/FXML file
- Adds `setupEmployeeUI()` / `submitEmployee()` methods following existing pattern

### D-06 — Delete Behavior
**Decision:** DELETE button for employees = **set status to INACTIVO** (soft archive).
- Does NOT call the hard-delete `DELETE /employees/{id}` endpoint
- Calls `PUT /employees/{id}` with `status: INACTIVO`

### D-07 — Password Change in EDIT Mode
**Decision (revised — out-of-phase temp password flow implemented):**
- A separate **"Cambiar contraseña"** button in the EDIT form calls `POST /employees/{id}/reset-password`.
- Backend generates a new `TmpXXXXXX` temp password, sets `mustChangePassword=true` on the employee, and returns it.
- Manager sees the temp password in a one-time dialog to communicate to the employee.
- Employee's next login forces a password change (handled by Phone/Tablet ChangePasswordScreen).

### D-08 — hireDate Field Widget
**Decision:** Use JavaFX **DatePicker** for the hireDate field.
- In the ADD form: pre-populated with **today's date** by default (user can change it)
- In the EDIT form: pre-populated with the employee's existing hireDate

### D-09 — Employee Search Popup Text
**Decision:** The `displayTextExtractor` for employees shows `"Nombre Apellido (username)"`.
- Example: `"Juan García (jgarcia)"`
- Allows managers to distinguish employees with identical names

### D-10 — Roles Empty Validation
**Decision:** If the user clicks Submit with **no role selected**, apply `blank-element` CSS class to the VBox container holding the role ToggleButtons (same red-border style as existing blank TextFields/ComboBoxes).
- No separate error dialog — consistent visual feedback
- Remove style on click inside the roles container (follow `addClickHandler` pattern)

### D-11 — ADD Form Grid Layout
**Decision:** 2-column GridPane, fields arranged as:

| Row | Col 0 | Col 1 |
|-----|-------|-------|
| 0 | name (Nombre) | lastname (Apellido) |
| 1 | email (span 2 cols) | ← |
| 2 | phoneNumber (opcional) | hireDate (DatePicker) |
| 3 | roles label + ToggleButtons (span 2 cols) | ← |

EDIT form adds `findItem` search row at Row 0 (existing items shift down by 1).

## Agent's Discretion

- Employee list presentation: JavaFX `ListView` with `EmployeeResponse` items, filter by name with existing `FilteredList`+`Popup` pattern from DataOperationController (matches `setupDynamicList` approach)
- ADD form fields (required): name, lastname, email, roles (toggle buttons), hireDate (DatePicker — see D-08)
- ADD form fields (optional): phoneNumber — **excluded from `verifyNotBlank()` check** (remains valid even when empty)
- Status field in ADD form: **not shown** — always ACTIVO on creation (confirmed GA-03)
- `username` field: auto-generated by backend on create; display as read-only Label in EDIT form
- Confirmation dialog text: `"Confirme la creación del empleado"` / `"Confirme la modificación del empleado"` / `"¿Desactivar este empleado?"` per action
- Error handling: `CustomDialog.showError()` for API failures (email already taken, etc.)
- After any submit: refresh the employee list and clear the form fields

## Codebase Context (at time of discussion)

### Backend (no changes needed for this phase)
- `GET /employees` → `List<EmployeeResponse>`
- `POST /employees` → `EmployeeRequest { name, lastname, roles(Set<Long>), phoneNumber, email, hireDate, password, status }`
- `PUT /employees/{id}` → same `EmployeeRequest` (partial: password only sent if changing)
- `EmployeeStatus`: ACTIVE / INACTIVE / SUSPENDED (Spanish names: ACTIVO/INACTIVO/SUSPENDIDO)
- `EmployeeResponse`: id, name, lastname, roles, phoneNumber, email, username, hireDate, status

### PC Frontend Pattern (from DataOperationController)
- Entity type selected via `setSelectedItem(String)` called by `DataController` when card is clicked
- `ToggleGroup` with ADD / EDIT / DELETE buttons → calls `handleEntityAction()`
- Dynamic form built programmatically in `GridPane` inside `dynamicFormVB`
- List shown as `ListView<T>` with `FilteredList` backing
- Submit via `submitButton` at bottom
- `CustomDialog.showError(msg)` for error feedback
- Services follow pattern: `EmployeeService.getAllEmployees()`, `EmployeeService.createEmployee(req)`, etc.

### New service needed
- `EmployeeService.java` in `com.yebur.service` (PC frontend) — mirrors `ProductService`, `CategoryService`
- `EmployeeResponse.java` in `com.yebur.model.response` (PC frontend)
- `EmployeeRequest.java` in `com.yebur.model.request` (PC frontend)
- Role loading: `RoleService.getAllRoles()` → `GET /roles`
